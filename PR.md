기능 구현에 앞서 설계 문서를 작성했습니다. 요구사항 정의부터 시퀀스 다이어그램, 클래스 다이어그램, ERD까지 개발 가능한 수준의 설계를 정리했습니다.
- **배경**: 브랜드, 상품, 좋아요, 주문 4개 도메인을 구현해야 하는데, 각 도메인 간 연관관계(브랜드→상품 연쇄 삭제, 좋아요 중복 방지, 주문 시 재고 원자적 차감)가 복잡하여 코드 작성 전 설계 합의가 필요했습니다.
- **목표**: 요구사항의 애매한 부분을 미결정 사항(Decision Record)으로 분리하고, 확정된 부분만으로 구현 가능한 설계 문서를 산출합니다.
- **결과**: 요구사항 명세서, 시퀀스 다이어그램(20개), 클래스 다이어그램, ERD를 작성했습니다. 확정할 수 없는 8개 의사결정은 별도 DEC 문서로 분리하여 개발 중 점진적으로 확정할 수 있도록 했습니다.

## Review Points

### 1. 계약에 의한 설계(Design by Contract)를 시퀀스 다이어그램에 적용

**배경 및 문제 상황:**
시퀀스 다이어그램이 "어떤 순서로 호출하는가"만 보여주면, 각 단계에서 **무엇이 보장되어야 하는지**가 드러나지 않습니다. 예를 들어 좋아요 등록에서 "상품이 존재해야 한다"는 사전조건과 "동일 사용자-상품 조합은 하나만 존재"라는 불변식이 다이어그램에 없으면, 구현 시 이를 놓치거나 잘못된 위치에서 검증할 수 있습니다.

**해결 방안:**
모든 시퀀스 다이어그램에 세 가지 계약 요소를 Note로 명시했습니다:

| 계약 요소 | 역할 | 예시 |
|---|---|---|
| **사전조건** | 이 연산을 호출하기 위해 만족해야 하는 조건 | 사용자 인증 완료, 대상 브랜드 존재 |
| **불변식** | 연산 전후로 항상 참이어야 하는 규칙 | 재고 >= 요청 수량, 브랜드 이름 필수 |
| **사후조건** | 연산이 성공했을 때 보장되는 결과 | 브랜드 및 연관 상품 모두 Soft Delete 완료 |

쓰기 작업에는 **트랜잭션 경계**도 함께 표기하여, 어디서부터 어디까지가 원자적 단위인지 시각적으로 드러나도록 했습니다.

**고민한 점:**
- 읽기 전용 조회(목록 조회, 상세 조회)에는 불변식이 없거나 미미하여, 사전조건과 사후조건만 간결하게 표기했습니다. 모든 다이어그램에 세 요소를 강제하면 오히려 노이즈가 됩니다.
- Public API(UC-B01 브랜드 정보 조회)는 인증이 필요 없으므로 사전조건 Note를 생략했습니다. "사전조건: 없음"이라고 적는 것은 정보가 아니라 형식이기 때문입니다.

### 2. 불변식의 소유자: 도메인 모델 vs DB

**배경 및 문제 상황:**
불변식을 **누가 보장하는가**에 따라 시퀀스 다이어그램의 participant 구성이 달라집니다. 모든 불변식을 도메인 모델에서 검증하면 일관성은 있지만, 어떤 경우에는 DB 레벨 제약조건이 더 효과적입니다.

**해결 방안:**
불변식의 성격에 따라 소유자를 다르게 결정했습니다:

| 불변식 | 소유자 | 이유 |
|---|---|---|
| 좋아요 중복 방지 (user_id + product_id) | **DB** (Unique Key) | 도메인 모델에 비즈니스 로직이 없음. DB 유니크 키로 원자적으로 보장하면 race condition도 방지 |
| 재고 >= 요청 수량 | **ProductModel** | 재고 차감은 도메인의 본질적 행위. 엔티티 메서드(`decreaseStock`)에서 검증 |
| 브랜드 이름 필수 | **BrandModel** (init block) | 엔티티 생성 시점의 유효성은 도메인 모델이 스스로 보장 |
| 브랜드 변경 불가 | **ProductModel** (update) | 상품의 브랜드 귀속은 도메인 규칙 |
| 주문 스냅샷 보존 | **OrderModel** | 주문 = 계약서라는 도메인 본질 |

**좋아요(Like)는 도메인 모델이 시퀀스 다이어그램에서 빠진 유일한 쓰기 도메인입니다.** 이는 설계 원칙(쓰기 작업에 Model 포함)의 예외가 아니라, 불변식의 소유자가 DB이기 때문에 도메인 모델을 거칠 필요가 없는 것입니다.

**고민한 점:**
- 좋아요 등록 시 "존재 확인 → 저장"을 서비스에서 두 단계로 수행하면, 두 요청이 동시에 존재 확인을 통과하고 모두 저장을 시도하는 race condition이 발생합니다. DB 유니크 키를 사용하면 INSERT 자체가 원자적이므로, 별도의 락 없이 멱등성을 보장할 수 있습니다.
- 반면 재고 차감처럼 "읽고 → 판단하고 → 쓰는" 연산은 DB 제약조건만으로는 부족합니다. 비관적 락으로 행을 잠근 뒤 도메인 모델에서 검증하는 방식이 필요하며, 이를 시퀀스 다이어그램에서 "상품 조회 (락 획득)"과 "비관적 락 획득"으로 명시했습니다.

### 3. 도메인 비즈니스 vs 어플리케이션 비즈니스 분리

**배경 및 문제 상황:**
DDD에서 "빈약한 도메인 모델을 피하라"는 원칙을 따르다 보면, 모든 로직을 엔티티에 넣으려는 유혹이 있습니다. 하지만 "username 중복 확인"이나 "브랜드 존재 확인"처럼 **다른 도메인/리포지토리와의 협력이 필요한 로직**까지 엔티티에 넣으면 엔티티가 인프라에 의존하게 됩니다.

**해결 방안:**
클래스 다이어그램에서 도메인 비즈니스와 어플리케이션 비즈니스를 명시적으로 분리했습니다:

- **도메인 비즈니스** (엔티티 메서드): 엔티티 자신의 상태만으로 판단할 수 있는 **변하지 않는 본질적 행위**
  - 재고 차감(`ProductModel.decreaseStock`), 총액 계산(`OrderModel.calculateTotalPrice`), 비밀번호 유효성 검증(`UserModel.init`)
- **어플리케이션 비즈니스** (서비스 오케스트레이션): 여러 도메인/리포지토리를 조율하는 **서비스마다 다를 수 있는 규칙**
  - 브랜드 존재 확인, 중복 좋아요 확인, 재고 확인 후 차감 조율

**고민한 점:**
- "주문 = 계약서"라는 은유를 적용했습니다. 주문의 본질은 "이 시점에 이 가격으로 이 수량을 계약했다"는 기록입니다. 따라서 `OrderItemModel`은 상품을 참조하는 것이 아니라 상품 정보를 **스냅샷으로 복사**합니다. 이는 상품 가격이 나중에 변경되어도 주문 당시 금액이 보존되어야 한다는 비즈니스 요구사항을 반영합니다.
- Like 도메인은 엔티티에 비즈니스 로직이 전혀 없는 유일한 도메인입니다. 좋아요의 본질이 "관계의 존재/부재"이므로, 도메인 모델이 가질 행위가 없습니다. 이를 억지로 Rich Domain Model로 만드는 것보다 DB 레벨 제약조건에 위임하는 것이 정직한 설계입니다.

### 4. 확정과 미확정의 명시적 분리: Decision Records

**배경 및 문제 상황:**
요구사항 분석 과정에서 즉시 확정할 수 없는 사항들이 발견되었습니다. 예를 들어 "좋아요 중복 요청 시 에러를 반환할 것인가, 멱등하게 무시할 것인가"는 사용자 경험과 API 설계 철학에 따라 달라지며, 설계 단계에서 단독으로 결정할 수 없습니다.

**해결 방안:**
미결정 사항을 요구사항 본문에 섞지 않고, 각각 독립된 Decision Record(DEC-001 ~ DEC-008)로 분리했습니다:

| ID | 제목 | 핵심 트레이드오프 |
|---|---|---|
| DEC-001 | 좋아요 중복 요청 처리 정책 | 에러 반환(명확) vs 멱등 무시(편의) |
| DEC-002 | 좋아요 삭제 전략 | Soft Delete(이력 보존) vs Hard Delete(Row explosion 방지) |
| DEC-003 | 타인 주문 조회 시 응답 코드 | 403(명확) vs 404(보안) |
| DEC-004 | 동시 주문 재고 제어 전략 | 비관적 락(단순) vs 낙관적 락(성능) vs 분산 락(확장) |
| DEC-005 | 주문 멱등성 보장 방식 | 멱등키 헤더 vs 클라이언트 UUID vs 서버 중복 감지 |
| DEC-006 | 상품 정렬 옵션 확장 범위 | likes_desc 정렬 시 집계 전략 (비정규화 vs 실시간 집계) |
| DEC-007 | 사용자 미존재 시 응답 코드 | **DECIDED**: 404 (명확한 에러 메시지 우선) |
| DEC-008 | 브랜드 삭제 시 연관 데이터 처리 범위 | 좋아요/주문 이력의 처리 범위 |

각 DEC 문서에는 **선택지 + 트레이드오프 + 영향도**를 정리하여, 개발 중 의사결정이 필요한 시점에 맥락을 다시 파악할 필요 없이 바로 결정할 수 있도록 했습니다.

**고민한 점:**
- 시퀀스 다이어그램과 요구사항 본문에서 미결정 사항이 등장하는 위치마다 DEC 문서 링크를 달았습니다. "이 부분은 아직 정해지지 않았다"는 것이 코드 리뷰처럼 명시적으로 보여야, 구현 시 임의로 결정하는 실수를 방지할 수 있습니다.
- DEC-002(좋아요 삭제 전략)는 클래스 다이어그램에도 영향을 줍니다. 현재 `LikeModel`은 Hard Delete를 전제로 `BaseEntity`를 상속하지 않도록 설계했으나, DEC-002가 Soft Delete로 결정되면 `BaseEntity` 상속과 `deleted_at` 컬럼 추가가 필요합니다. 이런 **설계 변경 파급 효과**를 DEC 문서에 명시해두었습니다.

### 5. ERD: FK 미사용과 삭제 전략의 물리적 반영

**배경 및 문제 상황:**
논리적으로 brands → products는 부모-자식 관계이지만, FK 제약조건을 걸면 데드락, 마이그레이션 어려움, 성능 저하 문제가 있습니다. 또한 도메인마다 삭제 전략(Soft Delete vs Hard Delete)이 다르므로, 이를 ERD에 반영해야 합니다.

**해결 방안:**
- **FK 미사용**: ERD의 관계선(`||--o{`)은 논리적 참조 관계를 나타내며, DB 레벨 FK가 아닙니다. 정합성은 어플리케이션 레벨에서 관리합니다.
- **likes 테이블**: `deleted_at` 없음 (Hard Delete 전제), `UNIQUE(user_id, product_id)` 제약조건으로 불변식을 DB 레벨에서 보장
- **order_items 테이블**: `product_name`, `price` 컬럼으로 상품 스냅샷을 물리적으로 저장. products 테이블을 JOIN하지 않아도 주문 시점 정보를 조회할 수 있습니다.

**고민한 점:**
- likes 테이블에 `updated_at`은 있지만 `deleted_at`은 없습니다. Hard Delete이므로 삭제 시 행 자체가 제거되어 `deleted_at`이 불필요합니다. 하지만 DEC-002 결정에 따라 Soft Delete로 변경될 수 있으므로, 이 점을 DEC 문서에 명시해두었습니다.
- `order_items.price`를 `bigDecimal`로 설정한 것은, 향후 할인/쿠폰 적용 시 소수점 금액이 발생할 수 있는 확장성을 고려한 것입니다.
