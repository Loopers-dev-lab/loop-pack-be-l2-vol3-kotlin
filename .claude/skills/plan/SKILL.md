요구사항/PRD: $ARGUMENTS

## 절차

1. `$ARGUMENTS`가 있으면 해당 요구사항 분석, 없으면 `PRD.md` 확인
2. 기존 `plan.md`가 있으면 진행 상황 파악 후 추가/수정 제안
3. 요구사항을 TDD로 구현할 수 있는 단위로 분해
4. 체크리스트 형태로 작성
5. 개발자 확인 후 저장

## plan.md 형식

```markdown
# 기능명

## 개요

요구사항 요약

## 구현 계획

### 1. 도메인 모델

- [ ] [P-A] 상품 Entity 필수 필드 검증
- [ ] [P-A] 상품 Entity 비즈니스 메서드
- [ ] [P-B] 주문 Entity 필수 필드 검증
- [ ] [P-B] 주문 Entity 비즈니스 메서드

### 2. 저장소

- [ ] [P-A] 상품 Repository 인터페이스 + 구현
- [ ] [P-B] 주문 Repository 인터페이스 + 구현

--- checkpoint: 도메인 + 저장소 검수 ---

### 3. 서비스

- [ ] [P-A] 상품 Service 핵심 로직
- [ ] [P-B] 주문 Service 핵심 로직

--- checkpoint: 서비스 레이어 검수 ---

### 4. API + E2E

- [ ] 주문 생성 API (상품+주문 통합)
- [ ] E2E 테스트

--- checkpoint: 전체 통합 검수 ---
```

## 표기법

### 병렬 그룹 `[P-X]`

같은 `[P-X]` 태그를 가진 항목은 독립적으로 동시 진행 가능.
이 표기는 **사람을 위한 신호** — "터미널 탭 하나 더 열어서 시켜도 꼬이지 않겠구나"의 판단 근거.

### 검수 단위 `--- checkpoint ---`

`--- checkpoint: 설명 ---` 구분선으로 검수 지점을 표시.
`/go`가 이 지점까지 연속 실행 후 종합 보고한다.

## 규칙

- 각 항목은 하나의 테스트로 검증 가능한 크기
- 도메인 → 저장소 → 서비스 → API 순서 (의존성 방향)
- 항목은 구체적으로 (예: "로그인 ID가 빈 값이면 회원가입 실패")
- 독립적인 도메인/기능은 `[P-X]` 태그로 병렬 그룹 표시
- 레이어 경계 또는 통합 지점에 `--- checkpoint ---` 배치
- 의존관계가 있는 항목은 같은 그룹 내에서 순서대로 배치
- 단일 도메인 작업이면 병렬 그룹 생략 가능 (강제하지 않음)
- 개발자 확인 없이 임의 생성 금지
