대상: $ARGUMENTS

기존 요구사항이 이미 정리된 상태에서, 새로운 라운드/제약/요구사항이 추가될 때 사용한다.
`/requirements-analysis`(백지 → 전체 분석)와 달리, 기존 문서를 기반으로 **변경분만** 분석한다.

## 절차

### 1️⃣ 기존 문서 파악

- `docs/requirements/` 디렉토리에서 기존 요구사항 문서를 읽는다.
- `docs/design/` 디렉토리에서 기존 설계 산출물(시퀀스, 클래스, ERD)을 읽는다.
- 현재 코드베이스의 도메인 구조를 파악한다.
- **기존 상태 요약**을 3~5줄로 정리하여 개발자에게 보여준다.

### 2️⃣ 새 요구사항과 기존 요구사항의 델타 분석

`$ARGUMENTS` 또는 개발자가 제공한 새 요구사항/제약을 분석한다.

변경 유형별로 분류:

| 유형 | 설명 | 예시 |
|------|------|------|
| **신규 도메인** | 기존에 없던 도메인 추가 | UserPoint, PointHistory |
| **도메인 확장** | 기존 도메인에 필드/규칙 추가 | Product에 재고 필드 추가 |
| **구조 변경** | 레이어/패턴/아키텍처 변경 | Service → Domain Service 통합 |
| **규칙 변경** | 비즈니스 규칙 수정/추가 | 주문 시 포인트 차감 추가 |
| **API 변경** | 엔드포인트 추가/수정 | 포인트 충전 API 신규 |

### 3️⃣ 애매한 부분을 질문으로 드러낸다

- 기존 결정과 충돌하는 부분을 명시한다.
- 새 요구사항에서 결정되지 않은 부분을 나열한다.
- `/requirements-analysis`와 동일한 질문 유형을 적용한다:
  - 정책 질문 / 경계 질문 / 확장 질문 / 동시성 질문
- 선택지가 있는 경우, 옵션 + 영향도 + **기존 구현과의 호환성**을 함께 제시한다.

### 4️⃣ 합의된 내용을 기존 문서에 증분 반영

새 요구사항 문서를 `docs/requirements/` 에 **별도 파일**로 생성한다.
- 파일명: `roundN-requirements-analysis.md` (N은 라운드 번호)
- 기존 문서는 수정하지 않는다 (이력 보존).

문서 구조는 `/requirements-analysis`의 산출물 구조를 따르되, 다음을 추가한다:

```markdown
## 0. 변경 요약 (Delta from Round N-1)

### 신규
- (새로 추가된 도메인, API, 규칙)

### 변경
- (기존 대비 달라진 점)

### 유지
- (변경 없이 그대로 사용하는 기존 구현)
```

### 5️⃣ 기존 설계 산출물과의 정합성 확인

- 새 요구사항이 기존 설계 산출물에 미치는 영향을 분석한다.
- 설계 문서 업데이트가 필요한 항목을 목록으로 제시한다:
  - 시퀀스 다이어그램: 새 흐름 추가 필요 여부
  - 클래스 다이어그램: 새 도메인/관계 추가 필요 여부
  - ERD: 새 테이블/컬럼 추가 필요 여부
- **직접 설계 문서를 수정하지 않는다** — 각 스킬(`/sequence`, `/class-diagram`, `/erd`)로 위임한다.

### 6️⃣ 잠재 리스크 업데이트

- 기존 리스크 중 해소된 것, 여전히 유효한 것을 구분한다.
- 새 요구사항으로 인해 추가된 리스크를 명시한다.

## 톤 & 스타일

- "전부 다시 정리하겠습니다"가 아니라 "이전 대비 이것이 달라졌습니다" 톤 유지
- 기존 결정을 존중하고, 변경이 필요한 부분만 제안한다
- 변경의 이유와 영향 범위를 항상 함께 설명한다

## 규칙

- 기존 요구사항 문서를 덮어쓰지 않는다 (별도 파일 생성)
- 델타 분석 없이 전체를 다시 작성하지 않는다
- 기존 결정과 충돌하는 새 요구사항은 반드시 개발자에게 확인한다
- 설계 산출물 업데이트는 직접 하지 않고 필요 목록만 제시한다
