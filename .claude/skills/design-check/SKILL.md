구현 또는 변경 계획을 세울 때 `docs/design/` 설계 문서를 먼저 확인하세요.

## 트리거 조건

- 새 도메인 구현 시
- 기존 기능 변경 시
- 버그 수정으로 도메인 로직이 영향받을 때
- 계획 수립(Plan Mode) 진입 전

## 확인 절차

### 1. 관련 도메인 식별

작업 대상 도메인을 먼저 파악한다. (예: product, order, like, brand 등)

### 2. 설계 문서 4개를 순서대로 확인

| 문서 | 확인 항목 |
|------|-----------|
| `docs/design/01-requirements.md` | 해당 도메인의 API 스펙, 비즈니스 규칙(BR/PR/LK/OR), 제약조건 |
| `docs/design/02-sequence-diagrams.md` | 관련 시퀀스 흐름, 레이어 간 호출 순서, 트랜잭션 경계 |
| `docs/design/03-class-diagram.md` | 엔티티 필드, 메서드 시그니처, 레이어 구조, 의존 관계 |
| `docs/design/04-erd.md` | 테이블 스키마, 컬럼 타입, 인덱스, FK 제약조건 |

### 3. 제약사항 요약 및 공유

설계 문서에서 추출한 내용을 아래 형식으로 개발자에게 공유한다.

```
## 설계 문서 확인 결과

**대상 도메인**: {도메인명}

### 비즈니스 규칙
- {01-requirements에서 추출한 규칙 목록}

### 시퀀스 흐름
- {02-sequence-diagrams에서 추출한 주요 흐름}

### 클래스 구조
- {03-class-diagram에서 추출한 엔티티/서비스 구조}

### DB 스키마
- {04-erd에서 추출한 테이블/제약조건}

### 구현 시 주의사항
- {설계 문서 기반으로 도출한 주의사항}
```

### 4. 설계와 구현 계획 정합성 검증

- 계획이 설계 문서와 어긋나는 부분이 있으면 개발자에게 알린다.
- 설계 문서에 없는 새로운 요소가 필요하면 개발자에게 확인한다.
- 정합성이 확인된 후에만 구현을 진행한다.

## 주의사항

- 설계 문서에 해당 도메인 정보가 없으면 그 사실을 개발자에게 알린다.
- 설계 문서 업데이트가 필요한 경우, 구현 전에 문서 수정 여부를 개발자에게 확인한다.
- 이 스킬은 계획 수립 단계에서 사용하며, 실제 코드 작성은 하지 않는다.
