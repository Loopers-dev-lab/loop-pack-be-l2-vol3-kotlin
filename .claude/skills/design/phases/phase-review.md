# phase-review: 설계 리뷰 + 문서 검증

전체 설계 문서를 리뷰하고, 코드베이스가 있으면 문서-코드 정합성도 확인한다.
리뷰 결과는 인라인으로 출력한다 (별도 파일 저장 없음).

---

## 사전 준비

존재하는 설계 문서를 모두 Read한다:
- `docs/design/01-requirements.md`
- `docs/design/02-sequence-diagrams.md`
- `docs/design/03-class-diagram.md`
- `docs/design/04-erd.md`

존재하지 않는 파일은 건너뛰고, 리뷰 보고서에 "문서 없음"으로 표시한다.

---

## Step 1: 문서별 리뷰

### 요구사항 (01-requirements.md)

- [ ] 유저 시나리오가 사용자 관점에서 작성되었는가
- [ ] 예외/조건 분기가 포함되었는가
- [ ] 유비쿼터스 언어가 정의되었는가
- [ ] 잠재 리스크가 식별되었는가
- [ ] 설계 결정 사항에 근거가 명시되었는가

### 시퀀스 다이어그램 (02-sequence-diagrams.md)

- [ ] 최소 2개 이상 존재하는가
- [ ] 책임 객체가 명확히 드러나는가
- [ ] 트랜잭션 경계가 표시되었는가
- [ ] Happy Path가 명확하고, 예외 흐름은 요구사항 명세서에 위임되었는가
- [ ] Facade는 다중 서비스 조율 시에만 사용되었는가 (1:1 pass-through Facade 없음)
- [ ] 참여자 레벨이 통일되었는가 (Controller/Facade/Service/Repository만)

### 클래스 다이어그램 (03-class-diagram.md)

- [ ] Entity/VO 구분이 적절한가
- [ ] 비즈니스 메서드가 포함되었는가
- [ ] 의존 방향이 올바른가 (interfaces → application → domain)
- [ ] 한 객체에 책임이 과도하게 집중되지 않았는가
- [ ] VO 정리 테이블이 있는가

### ERD (04-erd.md)

- [ ] 클래스 다이어그램과 정합성이 맞는가
- [ ] 관계의 주인이 명확한가 (FK 위치)
- [ ] 인덱스가 필요한 컬럼이 식별되었는가
- [ ] soft delete(deleted_at), 상태 관리가 고려되었는가
- [ ] 반정규화 필드의 사유가 명시되었는가

## Step 2: 문서 간 정합성 확인

| 확인 항목 | 기준 | 결과 |
|----------|------|------|
| 시퀀스 참여자 → 클래스에 존재 | 02 → 03 | ✅/❌ |
| 클래스 Entity 관계 → ERD에 반영 | 03 → 04 | ✅/❌ |
| 도메인 용어 일관성 | 01의 유비쿼터스 언어 | ✅/❌ |
| API 명세 → 시퀀스에 반영 | 01 → 02 | ✅/❌ |

---

## Step 3: 코드 정합성 확인 (선택적)

코드베이스가 있으면 (기존 구현이 있는 경우) verify-docs 절차를 따른다:

1. `apps/` 또는 `src/` 디렉토리가 존재하는지 확인한다
2. 존재하면 문서에 명시된 클래스명, 엔드포인트, 패턴을 Grep으로 실제 코드와 대조한다
3. 불일치 항목을 다음 형식으로 보고한다:
   - ✅ 검증 통과: 코드에서 확인됨
   - ❌ 불일치: 문서에는 X, 코드에서는 Y
   - ⚠️ 누락: 문서에 명시되었으나 코드에서 찾을 수 없음
   - 📝 미반영: 코드에 존재하나 문서에 기술되지 않음

코드베이스가 없으면 (신규 설계) 이 단계를 건너뛴다.
