# Phase Refactor: 구조 개선 (Tidy First)

TDD Refactor Phase. 모든 테스트가 통과하는 상태에서 구조적 변경만 수행한다.
동작 변경 절대 금지. 테스트 실패 시 즉시 되돌린다.

## 절차

1. **전체 테스트 통과 확인**: 통과 상태가 아니면 즉시 중단하고 보고한다.
   ```bash
   ./gradlew :apps:commerce-api:test
   ```

2. **개선 대상 식별**: 방금 Green Phase에서 작성한 코드를 중심으로 검토한다.
   개선 없이 깔끔하면 "구조 개선 불필요"로 보고하고 완료한다.

3. **한 번에 하나의 구조적 변경**만 수행한다:

   **프로덕션 코드:**
   - 중복 제거
   - 네이밍 개선 (의도를 명확히 표현)
   - 메서드 추출 (단일 책임)
   - 가드 절 도입 (early return)
   - 설명 변수 도입
   - 코드 이동 (레이어 경계 정리)
   - unused import 제거

   **테스트 코드:**
   - 공통 Arrange를 `@BeforeEach`로 추출
   - 테스트 픽스처/헬퍼 추출
   - `@Nested` 구조 재정리
   - 테스트명 개선 (의도 명확화)
   - 중복 시나리오 제거

4. **린트 수정 및 체크**:
   ```bash
   ./gradlew :apps:commerce-api:ktlintFormat
   ./gradlew :apps:commerce-api:ktlintCheck
   ```

5. **전체 테스트 재확인**: 실패하면 변경을 즉시 되돌린다.
   ```bash
   ./gradlew :apps:commerce-api:test
   ```

6. **보고**: Phase 완료 보고 형식으로 결과를 보고한다.

## 규칙

- **구조적 변경만** — 동작 변경 절대 금지.
- 한 번에 하나의 변경만 수행한다.
- 행위적 변경이 필요하다면 별도 작업으로 제안하고 이 Phase에서는 하지 않는다.
- 테스트 실패 시 변경을 되돌리고 무엇이 실패했는지 보고한다.
- 인접한 코드, 주석, 포맷을 "개선"하지 않는다 — 요청 범위만 변경한다.

## 완료 보고

```
## Refactor Phase 완료

- **개선 사항**: 수행한 구조적 변경 목록 (없으면 "구조 개선 불필요")
- **변경 파일**: 변경된 파일 목록 (경로 포함)
- **결과**: 전체 테스트 통과
- **다음 단계**: 사이클 완료 또는 "추가 요구사항이 있으면 다음 사이클 Red Phase로"
```
