PR 템플릿 기반으로 Pull Request 본문을 작성한다.

## 절차

1. 현재 브랜치의 변경 사항을 분석한다:
    - `git log main..HEAD --oneline`으로 커밋 이력을 확인한다
    - `git diff main...HEAD --stat`으로 변경된 파일 목록을 확인한다
    - `git diff main...HEAD`으로 전체 변경 내용을 확인한다
2. 변경된 주요 파일들을 읽어 구현 내용을 파악한다
3. `.github/pull_request_template.md` 템플릿을 기반으로 PR 본문을 작성한다
4. 작성된 PR 본문을 개발자에게 보여주고 확인받는다

## 작성 규칙

### Summary
- 첫 문단에 전체 작업을 1~2문장으로 요약한다
- **주요 구현 내용:** 불릿 리스트로 핵심 구현 사항을 나열한다 (각 항목은 **굵은 키워드**: 설명 형식)
- **구현된 기능:** 구현된 API 엔드포인트나 기능 목록을 나열한다

### Context & Decision → Review Points 로 변환
- 템플릿의 Context & Decision 섹션 대신 **�� Review Points** 섹션을 작성한다
- 리뷰어가 중점적으로 봐야 할 설계 결정 사항을 3~5개 선정한다
- 각 리뷰 포인트는 아래 구조를 따른다:

```
#### N. 리뷰 포인트 제목

**배경 및 문제 상황:**
왜 이 설계/구현이 필요했는지 문제 상황을 구체적으로 설명한다.
단순히 "~해야 했다"가 아니라 발생 가능한 문제와 그 영향을 명확히 기술한다.

**해결 방안:**
선택한 해결책을 설명한다. 다른 대안이 있었다면 비교하고, 왜 이 방식을 선택했는지 근거를 제시한다.

**구현 세부사항:** (필요시)
구현의 핵심 흐름이나 단계를 설명한다. 번호 매긴 목록이나 구조도로 작성할 수 있다.

**관련 코드:**
핵심 코드 스니펫을 포함한다. 파일 경로와 줄 번호를 명시한다.
코드 블록은 실제 구현 코드를 기반으로 작성한다 (가상의 코드 금지).

**고민한 점:**
- 트레이드오프, 제약사항, 향후 개선 가능성을 기술한다
- "~할 수도 있지만 ~이유로 ~를 선택했다" 형식으로 의사결정 과정을 드러낸다
- 리뷰어에게 검토를 요청하는 포인트를 명시한다

---
```

### Design Overview → ✅ Checklist 로 변환
- 템플릿의 Design Overview 섹션 대신 **✅ Checklist** 섹션을 작성한다
- 구현한 기능/작업 항목을 카테고리별로 그룹핑한다
- 각 항목은 `- [x]` 체크리스트 형식으로 작성한다
- 각 항목에 관련 파일 경로를 포함한다

```
### 카테고리명

- [x] **체크리스트 항목 제목**
  - 상세 설명
  - `파일/경로/파일명.kt` (줄 번호)
```

### Flow Diagram
- 복잡한 흐름이 있을 경우에만 포함한다
- Mermaid sequenceDiagram 또는 flowchart 형식으로 작성한다
- 텍스트 기반 구조도(`→`, `├─`, `└─`)를 활용할 수도 있다

### References
- 참고 자료가 있으면 작성하고, 없으면 섹션을 제거한다

## 작성 스타일

- **"왜"를 중심으로 작성한다**: 무엇을 했는지(What)보다 왜 그렇게 했는지(Why)가 핵심
- **문제 → 해결 → 고민 흐름**: 각 리뷰 포인트는 이 흐름을 일관되게 유지한다
- **실제 코드 기반**: 관련 코드 섹션에는 반드시 실제 구현 코드를 사용한다
- **구체적인 파일 경로**: 모든 코드 참조에 파일 경로를 명시한다
- **한국어로 작성**: 모든 설명, 제목, 고민 포인트를 한국어로 작성한다
- **대안 비교**: 가능하면 고려한 대안과 선택 이유를 함께 기술한다
- **트레이드오프 명시**: 선택한 방식의 장단점을 솔직하게 기술한다

## PR 제목 규칙

- 형식: `[volume-n] 작업 내용 요약`
- 예: `[volume-1] 회원가입, 로그인 구현`

## 최종 출력

작성된 PR 본문을 마크다운으로 출력한다. 개발자가 검토 후 수정할 수 있도록 한다.
