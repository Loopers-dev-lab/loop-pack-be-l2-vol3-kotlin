## Phase 1: Coupon 도메인 모델링

### 목표

쿠폰 템플릿(Admin이 생성)과 발급된 쿠폰(사용자가 소유)을 독립된 도메인으로 모델링한다.
정액/정률 할인 계산 로직은 도메인 모델에 캡슐화한다.

### 도메인 개념 정리

| 개념 | 분류 | 근거 |
|------|------|------|
| `CouponModel` | Entity | 쿠폰 템플릿. Admin이 생성하고, 할인 규칙을 정의하는 객체 |
| `CouponIssueModel` | Entity | 발급된 쿠폰. 사용자가 소유하고, 사용 상태가 변화하는 객체 |
| `CouponType` | Enum | 할인 방식 구분 (FIXED: 정액, RATE: 정률) |
| `CouponIssueStatus` | Enum | 발급 쿠폰 상태 (AVAILABLE, USED, EXPIRED) |

### 설계 의사결정: 쿠폰 템플릿과 발급 쿠폰을 분리한 이유

하나의 쿠폰 템플릿에서 여러 사용자에게 발급할 수 있다.
템플릿(할인 규칙)과 발급 인스턴스(소유자, 사용 상태)는 생명주기가 다르므로 별도 Entity로 분리한다.

### 파일 구조

```
domain/coupon/
  ├── CouponModel.kt          — 쿠폰 템플릿 Entity
  ├── CouponIssueModel.kt     — 발급 쿠폰 Entity
  ├── CouponType.kt           — FIXED, RATE
  ├── CouponIssueStatus.kt    — AVAILABLE, USED, EXPIRED
  ├── CouponRepository.kt     — 쿠폰 템플릿 Repository 인터페이스
  ├── CouponIssueRepository.kt — 발급 쿠폰 Repository 인터페이스
  ├── CouponService.kt        — 쿠폰 템플릿 Domain Service
  └── CouponIssueService.kt   — 발급 쿠폰 Domain Service
```

### CouponModel 설계

```kotlin
@Entity
@Table(name = "coupon")
class CouponModel(
    name: String,
    type: CouponType,        // FIXED | RATE
    value: Long,             // 정액: 할인 금액(원), 정률: 퍼센트(%)
    minOrderAmount: Long?,   // 최소 주문 금액 조건 (nullable)
    expiredAt: ZonedDateTime,
) : BaseEntity() {
    // 필드 정의 (protected set)

    init {
        validateName(name)
        validateValue(type, value)
    }

    /** 할인 금액을 계산한다. */
    fun calculateDiscount(orderAmount: Long): Long {
        validateMinOrderAmount(orderAmount)
        return when (type) {
            CouponType.FIXED -> value.coerceAtMost(orderAmount) // 주문 금액 초과 방지
            CouponType.RATE -> (orderAmount * value / 100)
        }
    }

    fun isExpired(): Boolean = ZonedDateTime.now().isAfter(expiredAt)

    private fun validateMinOrderAmount(orderAmount: Long) {
        minOrderAmount?.let {
            if (orderAmount < it) {
                throw CoreException(ErrorType.BAD_REQUEST,
                    "최소 주문 금액 조건을 충족하지 않습니다. (최소: ${it}원, 주문: ${orderAmount}원)")
            }
        }
    }
}
```

### CouponIssueModel 설계

```kotlin
@Entity
@Table(name = "coupon_issue")
class CouponIssueModel(
    couponId: Long,
    userId: Long,
    status: CouponIssueStatus = CouponIssueStatus.AVAILABLE,
) : BaseEntity() {
    // 필드 정의 (protected set)

    /** 쿠폰을 사용 처리한다. 이미 사용됐거나 만료된 쿠폰이면 예외. */
    fun use() {
        if (status != CouponIssueStatus.AVAILABLE) {
            throw CoreException(ErrorType.BAD_REQUEST,
                "사용할 수 없는 쿠폰입니다. (현재 상태: $status)")
        }
        this.status = CouponIssueStatus.USED
    }

    fun isUsable(): Boolean = status == CouponIssueStatus.AVAILABLE
}
```

## Phase 6: Order + Coupon 통합

### 목표

주문 시 쿠폰을 적용해 할인받을 수 있도록 주문 흐름을 확장한다.

### 변경 사항

#### OrderModel 필드 추가

```kotlin
// 기존 필드에 추가
@Column(name = "coupon_issue_id")
var couponIssueId: Long? = null

@Column(name = "original_total_amount", nullable = false)
var originalTotalAmount: Long = 0       // 쿠폰 적용 전 금액

@Column(name = "discount_amount", nullable = false)
var discountAmount: Long = 0            // 쿠폰 할인 금액

// 기존 totalAmount → 최종 결제 금액으로 사용
```

#### 주문 요청에 couponId 추가

```json
{
  "items": [
    { "productId": 1, "quantity": 2 },
    { "productId": 3, "quantity": 1 }
  ],
  "couponId": 42
}
```

`couponId`는 nullable — 미적용 시 생략 가능.

#### 주문 흐름 (OrderFacade)

```
1. 주문 요청
2. 상품 조회 + 재고 확인
3. couponId가 있으면:
   a. 발급 쿠폰 조회 (CouponIssueModel)
   b. 소유자 검증 (본인 쿠폰인지)
   c. 사용 가능 여부 검증 (AVAILABLE + 만료 안 됨)
   d. 쿠폰 템플릿 조회 → 할인 금액 계산
   e. 쿠폰 사용 처리 (USED)
4. 재고 차감
5. 주문 엔티티 생성 (원금, 할인, 최종 금액 포함)
6. 저장
```

### 쿠폰 적용 규칙 (Quest 명세)

- 주문 1건당 쿠폰 1장만 적용
- 존재하지 않거나, 이미 사용됐거나, 만료됐거나, 타 유저 소유 쿠폰 → 주문 실패
- 주문 성공 시 쿠폰은 즉시 USED 상태로 변경, 재사용 불가
- 주문 스냅샷에 쿠폰 적용 전 금액, 할인 금액, 최종 결제 금액 모두 포함

### TDD 시나리오

| 시나리오 | 기대 결과 |
|---------|-----------|
| 쿠폰 없이 주문 | 기존과 동일하게 동작 |
| 정액 쿠폰 적용 주문 | 할인 금액 반영, 쿠폰 USED 전환 |
| 정률 쿠폰 적용 주문 | 할인 금액 반영, 쿠폰 USED 전환 |
| 이미 사용된 쿠폰으로 주문 | `CoreException(BAD_REQUEST)` |
| 만료된 쿠폰으로 주문 | `CoreException(BAD_REQUEST)` |
| 타 유저 쿠폰으로 주문 | `CoreException(BAD_REQUEST)` |
| 존재하지 않는 쿠폰으로 주문 | `CoreException(NOT_FOUND)` |

### 완료 기준

- [ ] 쿠폰 적용 주문이 정상 동작
- [ ] 주문 스냅샷에 원금/할인/최종 금액이 모두 포함
- [ ] 쿠폰 사용 불가 케이스에서 주문 실패 확인
- [ ] 테스트 통과

## Phase 7: Lock 적용

### 목표

동시성 이슈가 발생할 수 있는 구간에 적절한 Lock 전략을 적용한다.

### Lock 전략 결정

| 대상 | 전략 | 근거 |
|------|------|------|
| 재고 차감 | 비관적 락 (`SELECT ... FOR UPDATE`) | 정확한 값 기반 연산 필수. 동시 주문 시 순차 처리하여 음수 재고 방지 |
| 쿠폰 사용 | 비관적 락 (`SELECT ... FOR UPDATE`) | 1회 사용 제약 필수. 동시 요청 시 선착순 1명만 성공 |
| 좋아요 수 | 낙관적 락 (`@Version`) + 재시도 | 충돌 빈도가 낮고, 실패해도 재시도로 해결 가능 |

### 구현 방식

#### 재고 차감 — 비관적 락

```kotlin
// ProductJpaRepository
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM ProductModel p WHERE p.id IN :ids AND p.deletedAt IS NULL")
fun findAllByIdsForUpdate(@Param("ids") ids: List<Long>): List<ProductModel>
```

OrderFacade에서 상품 조회 시 `findAllByIdsForUpdate`를 사용하면,
트랜잭션이 끝날 때까지 다른 트랜잭션이 해당 상품 행을 수정할 수 없다.

#### 쿠폰 사용 — 비관적 락

```kotlin
// CouponIssueJpaRepository
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT ci FROM CouponIssueModel ci WHERE ci.id = :id AND ci.deletedAt IS NULL")
fun findByIdForUpdate(@Param("id") id: Long): CouponIssueModel?
```

#### 좋아요 수 — 낙관적 락

```kotlin
// ProductModel에 @Version 추가
@Version
@Column(name = "version")
var version: Long = 0
    protected set
```

LikeFacade에서 likesCount 변경 시 `OptimisticLockingFailureException`이 발생하면
재시도 로직으로 처리한다.

### 트랜잭션 경계

```
OrderFacade.createOrder()  ← @Transactional 경계
  ├─ 상품 조회 (비관적 락으로 행 잠금)
  ├─ 쿠폰 조회 + 검증 + 사용 처리 (비관적 락)
  ├─ 재고 차감
  ├─ 주문 생성 + 저장
  └─ 하나라도 실패 → 전체 롤백
```

## Phase 8: 동시성 테스트

### 목표

동시성 이슈가 발생할 수 있는 3가지 시나리오를 `CountDownLatch` + `ExecutorService`로 검증한다.

### 테스트 시나리오

#### 테스트 1: 좋아요 수 정합성

```
조건: 상품 1개, 10명이 동시에 좋아요 등록
검증: product.likesCount == 10
```

#### 테스트 2: 쿠폰 단일 사용 보장

```
조건: 발급 쿠폰 1장, 여러 기기에서 동시에 주문 시도
검증: 성공한 주문 == 1건, 쿠폰 상태 == USED
```

#### 테스트 3: 재고 정합성

```
조건: 재고 10개 상품, 15명이 동시에 1개씩 주문
검증: 성공한 주문 == 10건, 실패한 주문 == 5건, 최종 재고 == 0
```

#### 테스트 4: 트랜잭션 롤백

```
조건: 재고 충분 + 사용 불가 쿠폰으로 주문
검증: 주문 실패, 재고 원복, 쿠폰 상태 변경 없음
```

### 테스트 구조 예시

```kotlin
@Test
fun `동시에 10명이 주문할 때 재고가 정확히 차감된다`() {
    val threadCount = 10
    val latch = CountDownLatch(threadCount)
    val executor = Executors.newFixedThreadPool(threadCount)
    val successCount = AtomicInteger(0)

    repeat(threadCount) {
        executor.submit {
            try {
                orderFacade.createOrder(userId = it.toLong(), items = listOf(...))
                successCount.incrementAndGet()
            } catch (e: Exception) {
                // 재고 부족으로 실패
            } finally {
                latch.countDown()
            }
        }
    }

    latch.await()

    val product = productRepository.findById(productId)
    assertThat(product.stockQuantity).isGreaterThanOrEqualTo(0)
    assertThat(successCount.get()).isEqualTo(expectedSuccessCount)
}
```
