## 1. 변경된 파일 전체 목록과 작업 목적

커밋되지 않은 파일은 크게 세 가지 범주로 나뉜다.

### 1-1. 기존 파일 수정 (15개)

이미 존재하던 파일을 수정한 것으로, 주문 흐름에 쿠폰 적용을 통합하고 동시성 제어를 적용하기 위한 변경이다.

| 파일 경로 | 변경 목적 |
|----------|----------|
| `application/like/LikeFacade.kt` | 좋아요 등록/취소 시 상품의 좋아요 수를 변경할 때 **낙관적 락 충돌에 대비한 재시도 로직**(최대 3회)을 추가했다. `ObjectOptimisticLockingFailureException`이 발생하면 자동으로 재시도한다. |
| `application/order/OrderFacade.kt` | 주문 생성 메서드에 **`@Transactional`을 선언**하여 주문 전체 흐름의 원자성을 보장한다. 쿠폰 검증(소유자 확인, 사용 가능 여부, 만료 여부)과 쿠폰 할인 적용 로직을 추가했다. 상품 조회 시 **비관적 락(`findAllByIdsForUpdate`)** 을 사용하여 재고 동시 차감 문제를 방지한다. 쿠폰 사용 처리 시에도 **비관적 락(`findByIdForUpdate`)** 을 사용하여 동시 사용을 방지한다. |
| `application/order/OrderInfo.kt` | 주문 응답 정보에 쿠폰 관련 필드를 추가했다. `couponIssueId`(적용된 쿠폰 발급 번호), `originalTotalAmount`(쿠폰 적용 전 원래 금액), `discountAmount`(할인 금액), `totalAmount`(최종 결제 금액)을 포함한다. |
| `domain/order/OrderModel.kt` | 주문 도메인 엔티티에 쿠폰 할인 관련 필드(`couponIssueId`, `originalTotalAmount`, `discountAmount`)를 추가했다. `applyCouponDiscount()` 메서드로 할인 금액을 반영하고, `recalculateTotalAmount()` 메서드가 원래 금액에서 할인 금액을 뺀 최종 금액을 계산한다(최솟값 0원 보장). |
| `domain/order/OrderRepository.kt` | 날짜 범위로 주문을 조회하는 메서드(`findAllByUserIdAndCreatedAtBetween`)를 추가했다. 대고객 주문 목록 조회 API에서 시작일과 종료일을 기준으로 필터링할 때 사용한다. |
| `domain/order/OrderService.kt` | 주문 도메인 서비스에 날짜 범위 조회(`findByUserIdAndDateRange`)와 사용자별 주문 조회(`findByIdAndUserId`) 메서드를 추가했다. 조회 메서드에는 `@Transactional(readOnly = true)`를, 생성 메서드에는 `@Transactional`을 선언했다. |
| `domain/product/ProductModel.kt` | 상품 엔티티에 **`@Version` 필드를 추가**하여 낙관적 락을 활성화했다. 좋아요 수 변경 시 JPA가 자동으로 버전을 비교하여 동시 수정 충돌을 감지한다. `decreaseStock()`과 `increaseStock()` 메서드로 재고를 관리하며, 재고 부족 시 예외를 발생시킨다. |
| `domain/product/ProductRepository.kt` | **비관적 락을 사용하는 조회 메서드**(`findAllByIdsForUpdate`)를 인터페이스에 추가했다. 주문 생성 시 상품 행을 잠그기 위해 사용한다. |
| `domain/product/ProductService.kt` | 비관적 락 조회를 위임하는 `findAllByIdsForUpdate()` 메서드를 추가했다. 좋아요 수 증감 메서드에 `@Transactional`을 선언하여 낙관적 락이 정상 동작하도록 했다. |
| `infrastructure/auth/JwtAuthenticationFilter.kt` | 인증이 필요한 API 경로 목록에 쿠폰(`/api/v1/coupons/`), 사용자 쿠폰 조회(`/api/v1/users/me/`), 좋아요(`/likes` suffix) 경로를 추가했다. 이 경로들로 들어오는 요청은 JWT 토큰 검증을 거친다. |
| `infrastructure/order/OrderJpaRepository.kt` | 날짜 범위 조회를 위한 JPQL 커스텀 쿼리를 추가했다. `CAST(o.createdAt AS localdate)`를 사용하여 시간대(timezone)를 고려한 날짜 비교를 수행한다. |
| `infrastructure/order/OrderRepositoryImpl.kt` | 날짜 범위 조회 메서드의 구현체를 추가했다. JPA Repository에 위임하는 어댑터 패턴이다. |
| `infrastructure/product/ProductJpaRepository.kt` | **`@Lock(LockModeType.PESSIMISTIC_WRITE)`** 어노테이션이 붙은 `findAllByIdsForUpdate()` 메서드를 추가했다. 이 메서드는 `SELECT ... FOR UPDATE` SQL을 생성하여 조회한 상품 행에 배타적 락을 건다. |
| `infrastructure/product/ProductRepositoryImpl.kt` | 비관적 락 조회 메서드를 JPA Repository에 위임하는 구현체를 추가했다. |
| `test/.../order/OrderFacadeTest.kt` | 주문 생성 테스트에 쿠폰 관련 시나리오를 추가했다. 정액 쿠폰 적용, 정률 쿠폰 적용, 이미 사용된 쿠폰으로 주문 시도, 만료된 쿠폰으로 주문 시도, 다른 사용자의 쿠폰으로 주문 시도, 존재하지 않는 쿠폰으로 주문 시도 등 6개 시나리오를 검증한다. |

### 1-2. 새로 추가된 파일 - 쿠폰 도메인 (16개 소스 + 8개 테스트)

쿠폰 기능을 처음부터 구현한 파일들이다. 구현-시나리오.md의 Phase 1~4에 해당한다.

#### 도메인 계층 (domain/coupon/)

| 파일 | 역할 |
|------|------|
| `CouponModel.kt` | **쿠폰 템플릿 엔티티**. 관리자가 생성하는 쿠폰의 원본 정의이다. 쿠폰 이름, 할인 유형(정액/정률), 할인 값, 최소 주문 금액 조건, 만료일을 가진다. `calculateDiscount(orderAmount)` 메서드가 할인 금액을 계산한다. 정액 쿠폰은 주문 금액을 초과하지 않도록 상한을 둔다. 정률 쿠폰은 주문 금액의 N%를 계산한다. |
| `CouponIssueModel.kt` | **발급된 쿠폰 엔티티**. 특정 사용자에게 발급된 쿠폰 인스턴스이다. 쿠폰 템플릿 번호, 사용자 번호, 상태(AVAILABLE/USED/EXPIRED)를 가진다. `use()` 메서드를 호출하면 AVAILABLE에서 USED로 상태가 전이된다. 이미 사용했거나 만료된 쿠폰에 `use()`를 호출하면 예외가 발생한다. 데이터베이스에 `(coupon_id, user_id)` 유니크 제약이 걸려 있어 동일 쿠폰의 중복 발급을 방지한다. |
| `CouponType.kt` | 쿠폰 할인 유형을 정의하는 열거형이다. `FIXED`(정액 할인)와 `RATE`(정률 할인) 두 가지가 있다. |
| `CouponIssueStatus.kt` | 발급된 쿠폰의 상태를 정의하는 열거형이다. `AVAILABLE`(사용 가능), `USED`(사용 완료), `EXPIRED`(만료) 세 가지이다. |
| `CouponRepository.kt` | 쿠폰 템플릿의 저장소 인터페이스이다. 저장, 단건 조회, 다건 조회(ID 목록), 페이징 조회 메서드를 정의한다. |
| `CouponIssueRepository.kt` | 발급 쿠폰의 저장소 인터페이스이다. 저장, 단건 조회, 사용자+쿠폰 조합 조회(중복 확인용), 사용자별 페이징 조회, 쿠폰별 페이징 조회, **비관적 락 조회(`findByIdForUpdate`)** 메서드를 정의한다. |
| `CouponService.kt` | 쿠폰 템플릿 도메인 서비스이다. 생성, 조회, 삭제(소프트 삭제) 기능을 제공한다. 조회 메서드에는 `@Transactional(readOnly = true)`, 쓰기 메서드에는 `@Transactional`을 선언했다. |
| `CouponIssueService.kt` | 발급 쿠폰 도메인 서비스이다. 쿠폰 발급(`issue`) 시 쿠폰 존재 여부, 만료 여부, 중복 발급 여부를 순서대로 검증한다. `DataIntegrityViolationException`(유니크 제약 위반)을 catch하여 동시 발급 시도에 대한 방어 처리를 한다. `findByIdForUpdate()`는 비관적 락으로 조회하여 동시 사용 방지에 사용한다. |

#### 어플리케이션 계층 (application/coupon/)

| 파일 | 역할 |
|------|------|
| `CouponAdminFacade.kt` | 관리자용 쿠폰 유스케이스를 조율하는 파사드이다. 쿠폰 템플릿 생성, 조회, 수정, 삭제, 발급 내역 조회를 처리한다. |
| `CouponAdminInfo.kt` | 관리자용 응답 정보 클래스이다. 쿠폰 템플릿 정보(`CouponAdminInfo`)와 발급 내역 정보(`CouponIssueAdminInfo`)를 정의한다. |
| `CouponFacade.kt` | 대고객 쿠폰 유스케이스를 조율하는 파사드이다. 쿠폰 발급과 내 쿠폰 목록 조회를 처리한다. 목록 조회 시 N+1 쿼리를 방지하기 위해 쿠폰 ID를 모아 한 번에 조회(bulk fetch)한다. |
| `CouponIssueInfo.kt` | 대고객 응답 정보 클래스이다. 발급 쿠폰의 상태와 쿠폰 템플릿 상세 정보를 결합한다. AVAILABLE 상태이지만 쿠폰 만료일이 지난 경우 EXPIRED로 표시하는 동적 만료 판정 로직이 포함되어 있다. |

#### 인프라스트럭처 계층 (infrastructure/coupon/)

| 파일 | 역할 |
|------|------|
| `CouponJpaRepository.kt` | 쿠폰 템플릿의 Spring Data JPA 저장소이다. 기본적인 조회와 페이징을 제공한다. |
| `CouponRepositoryImpl.kt` | 도메인 `CouponRepository` 인터페이스의 구현체이다. JPA 저장소에 위임하는 어댑터 패턴이다. |
| `CouponIssueJpaRepository.kt` | 발급 쿠폰의 Spring Data JPA 저장소이다. **`@Lock(LockModeType.PESSIMISTIC_WRITE)`가 선언된 `findByIdForUpdate()` 메서드**가 포함되어 있다. 이 메서드는 `SELECT ... FOR UPDATE` SQL을 생성하여 동시 쿠폰 사용을 방지한다. |
| `CouponIssueRepositoryImpl.kt` | 도메인 `CouponIssueRepository` 인터페이스의 구현체이다. JPA 저장소에 위임하는 어댑터 패턴이다. |

## 3. round4_Quest 체크리스트 충족 여부

round4_Quest 문서에서 정의한 체크리스트를 항목별로 점검한다.

### 3-1. Coupon 도메인

| 체크리스트 항목 | 충족 여부 | 구현 위치와 방식 |
|---------------|----------|----------------|
| 쿠폰은 사용자가 소유하고 있으며, 이미 사용된 쿠폰은 사용할 수 없어야 한다 | **충족** | `CouponIssueModel`이 `userId` 필드로 소유자를 기록한다. `use()` 메서드는 AVAILABLE 상태가 아니면 예외를 발생시킨다. `OrderFacade.validateCoupon()`에서 소유자 검증과 사용 가능 여부를 확인한다. |

## 4. 동시성 제어 전략 요약

구현에서 선택한 락 전략과 그 근거를 정리한다. 이 내용은 round4_Quest의 "과제 집중할 점"과 metoring 문서의 멘토링 내용을 반영한다.

### 4-1. 재고 차감 — 비관적 락 (Pessimistic Write Lock)

| 항목 | 내용 |
|------|------|
| 적용 위치 | `ProductJpaRepository.findAllByIdsForUpdate()` |
| SQL | `SELECT p FROM ProductModel p WHERE p.id IN :ids AND p.deletedAt IS NULL` + `FOR UPDATE` |
| 사용 시점 | `OrderFacade.createOrder()` 내부에서 상품 조회 시 |
| 선택 근거 | 재고는 정확한 값 기반 연산이 필요하다. 재고가 5개 남았을 때 동시에 3명이 주문하면, 순차적으로 처리하여 2명은 성공하고 1명은 재고 부족으로 실패해야 한다. 비관적 락은 행을 잠근 뒤 재고를 읽고 차감하므로 음수 재고가 발생하지 않는다. |
| 트레이드오프 | 비관적 락은 데이터베이스 커넥션을 오래 점유할 수 있다. metoring 문서에서 언급한 것처럼, 트래픽이 높아지면 커넥션 풀 고갈 위험이 있다. 개선 방향으로는 재고 테이블 분리, 원자적 UPDATE(DB 위임), 분산 락(Redis) 등이 있다. |

### 4-2. 쿠폰 사용 — 비관적 락 (Pessimistic Write Lock)

| 항목 | 내용 |
|------|------|
| 적용 위치 | `CouponIssueJpaRepository.findByIdForUpdate()` |
| SQL | `SELECT ci FROM CouponIssueModel ci WHERE ci.id = :id AND ci.deletedAt IS NULL` + `FOR UPDATE` |
| 사용 시점 | `OrderFacade.applyCoupon()` 내부에서 쿠폰 사용 처리 시 |
| 선택 근거 | 쿠폰은 1회만 사용 가능하다. 여러 기기에서 동시에 같은 쿠폰으로 주문할 때, 정확히 1명만 성공해야 한다. 비관적 락으로 쿠폰 행을 잠그면 나머지 요청은 대기하다가 쿠폰이 USED 상태임을 확인하고 실패한다. |
| 추가 방어 | `CouponIssueModel`의 데이터베이스 테이블에 `(coupon_id, user_id)` 유니크 제약이 걸려 있고, `CouponIssueService.issue()`에서 `DataIntegrityViolationException`을 catch하여 동시 발급 시도도 방지한다. |

### 4-3. 좋아요 수 — 낙관적 락 (Optimistic Lock) + 재시도

| 항목 | 내용 |
|------|------|
| 적용 위치 | `ProductModel`의 `@Version` 필드 |
| 재시도 로직 | `LikeFacade.retryOnOptimisticLock()` — 최대 3회 재시도 |
| 선택 근거 | 좋아요는 충돌 빈도가 상대적으로 낮고, 실패해도 재시도로 해결할 수 있다. 비관적 락으로 상품 행을 잠그면 상품 조회까지 영향을 받을 수 있으므로, 낙관적 락으로 충돌 시에만 재시도하는 것이 효율적이다. |
| 트레이드오프 | metoring 문서에서 "재시도하는 것이 낙관적 락이 아님! 실패해도 괜찮을 때 낙관적 락을 써야 함"이라고 언급되었다. 좋아요 수의 경우 일시적인 부정확도는 사용자 경험에 큰 영향을 미치지 않으므로 재시도 방식이 적절하다. |

## 5. 트랜잭션 경계 구조

주문 생성 흐름의 트랜잭션 경계를 시각적으로 표현한다.

```
OrderFacade.createOrder()  ← @Transactional 시작
  │
  ├─ 1. 중복 상품 검증 (저렴한 검증을 먼저 수행)
  │
  ├─ 2. 상품 조회 + 비관적 락 획득
  │      └─ productService.findAllByIdsForUpdate()
  │         └─ SELECT ... FOR UPDATE (상품 행 잠금)
  │
  ├─ 3. 쿠폰 사전 검증 (couponIssueId가 있는 경우)
  │      └─ validateCoupon(): 소유자, 사용 가능 상태 확인
  │         └─ 일반 조회 (락 없음, 빠른 실패를 위한 선행 검증)
  │
  ├─ 4. 브랜드 조회 (스냅샷용)
  │
  ├─ 5. 주문 생성 + 재고 차감
  │      └─ orderService.createOrder()
  │         └─ 각 상품에 대해 product.decreaseStock()
  │         └─ 주문 엔티티 + 주문 항목 엔티티 저장
  │
  ├─ 6. 쿠폰 할인 적용 (couponIssueId가 있는 경우)
  │      └─ applyCoupon()
  │         ├─ couponIssueService.findByIdForUpdate() ← 비관적 락
  │         ├─ couponIssue.use() ← AVAILABLE → USED
  │         ├─ coupon.calculateDiscount(orderAmount) ← 할인 금액 계산
  │         └─ order.applyCouponDiscount() ← 할인 반영
  │
  └─ 7. OrderInfo 반환
       └─ @Transactional 종료 → 커밋 (실패 시 전체 롤백)
```

## 8. 아키텍처 계층 일관성 점검

구현된 코드가 CLAUDE.md에서 정의한 레이어드 아키텍처 + DIP 원칙을 따르고 있는지 점검한다.

| 원칙 | 충족 여부 | 근거 |
|------|----------|------|
| Repository 인터페이스는 도메인 계층에 정의 | **충족** | `CouponRepository`, `CouponIssueRepository`가 `domain/coupon/` 패키지에 위치 |
| Repository 구현체는 인프라스트럭처 계층에 정의 | **충족** | `CouponRepositoryImpl`, `CouponIssueRepositoryImpl`이 `infrastructure/coupon/` 패키지에 위치 |
| Controller → Facade → Domain Service 흐름 | **충족** | `CouponV1Controller` → `CouponFacade` → `CouponIssueService`/`CouponService` |
| Application 계층은 경량으로 유지 | **충족** | `CouponFacade`, `CouponAdminFacade`는 도메인 서비스 호출을 조율할 뿐, 비즈니스 로직을 포함하지 않음 |
| 도메인 객체가 비즈니스 규칙을 캡슐화 | **충족** | `CouponModel.calculateDiscount()`, `CouponIssueModel.use()` 등 비즈니스 로직이 도메인 모델에 위치 |
| API Request/Response DTO와 Application Info DTO 분리 | **충족** | `CouponAdminV1Dto`(API 계층)와 `CouponAdminInfo`(Application 계층)가 분리됨 |
| 도메인 계층은 다른 계층에 의존하지 않음 | **충족** | `domain/coupon/` 패키지의 파일들은 Spring 프레임워크 어노테이션(`@Entity`, `@Component`)만 사용하며, 인프라스트럭처나 인터페이스 계층의 코드를 참조하지 않음 |

## 9. 결론

Round 4 요구사항의 핵심 목표인 **쿠폰 도메인 구현**, **주문 흐름 트랜잭션 적용**, **동시성 제어**는 모두 구현되었다.

- 쿠폰 도메인은 템플릿과 발급 쿠폰을 분리하여 모델링하고, 정액/정률 할인 로직을 도메인 모델에 캡슐화했다.
- 주문 흐름은 `@Transactional`로 원자성을 보장하고, 재고와 쿠폰에 비관적 락을 적용하여 동시성 문제를 해결했다.
- 좋아요에는 낙관적 락 + 재시도를 적용하여 성능과 정합성의 균형을 맞추었다.
- 동시성 테스트 4개 시나리오가 모두 구현되어 있고, 단위 테스트 57개가 도메인/서비스/파사드/인프라 전 계층을 커버한다.
- 미구현 상태였던 Brand, Product, Like, Order의 대고객/관리자 컨트롤러가 모두 완성되었다.

남은 작업은 `.http/` 테스트 파일 정리와 트랜잭션 분석 Skill 작성 확인이다.
