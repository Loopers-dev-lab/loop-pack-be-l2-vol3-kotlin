# 고객 API / 어드민 API 분리 전략

## 문제 인식

요구사항에 고객 API(`/api/v1`)와 어드민 API(`/api-admin/v1`)가 존재한다. 두 API는 인증 방식, 비즈니스 복잡도, 노출 정보가 모두 다르므로, 같은 코드 경로에서 처리하는 것은 부적절하다.

| 관점 | 고객 API | 어드민 API |
|------|----------|-----------|
| URL prefix | `/api/v1` | `/api-admin/v1` |
| 인증 | LoginId/LoginPw 헤더 | LDAP 헤더 |
| 비즈니스 복잡도 | 높음 (재고 차감, 멱등성, 스냅샷, 기간 필터) | 낮음 (단순 CRUD) |
| 노출 정보 | 고객용 최소 정보 | 관리 정보 포함 (재고, 생성일 등) |
| API 수 | 9개 | 12개 |

## 선택지 비교

### 패키지 레벨 분리 (같은 모듈)

```
apps/commerce-api/
├── interfaces/
│   ├── api/         ← 고객 API
│   └── api-admin/   ← 어드민 API
├── application/     ← Facade (필요시 고객/어드민 분리)
├── domain/          ← 공유
└── infrastructure/  ← 공유
```

| 관점 | 평가 |
|------|------|
| 도메인 공유 | ✅ 같은 Entity, Repository를 자연스럽게 공유 |
| 설정 복잡도 | ✅ 추가 모듈 설정 불필요 |
| 관심사 분리 | ✅ Interfaces/Application 레벨에서 분리 |
| 독립 배포 | ❌ 불가 (같은 프로세스) |

### 모듈 레벨 분리 (별도 앱)

```
apps/
├── commerce-api/        ← 고객용
├── commerce-admin-api/  ← 어드민용
```

| 관점 | 평가 |
|------|------|
| 도메인 공유 | ❌ domain 모듈을 별도로 추출해야 함 |
| 설정 복잡도 | ❌ 빌드/설정/테스트 중복 |
| 관심사 분리 | ✅ 물리적 분리 |
| 독립 배포 | ✅ 가능 |

## 결론: 패키지 레벨 분리

### 선택 근거

1. **어드민 API는 단순 CRUD** — Facade가 도메인 서비스에 직접 위임하면 충분하다. 별도 모듈로 분리할 만큼의 독립적인 비즈니스 로직이 없다
2. **도메인은 공유** — Brand, Product, Order는 고객/어드민이 같은 엔티티를 사용한다. 모듈 분리 시 domain 모듈 추출이 필요하고, 이는 DIP 강제 전략에서 논의한 "JPA Entity = Domain Entity" 충돌 문제를 다시 발생시킨다
3. **독립 배포가 당장 필요하지 않음** — 현재 요구사항에 독립 배포 요건이 없다
4. **인증 분리는 URL prefix로 해결** — `/api/v1/*`과 `/api-admin/v1/*`에 각각 다른 인증 필터를 적용하면 깔끔하게 분리된다

### 구현 규칙

- **Interfaces 계층**: `interfaces/api/`(고객)와 `interfaces/api-admin/`(어드민)으로 패키지 분리
- **Application 계층**: 어드민 Facade는 도메인 서비스에 단순 위임. 복잡한 오케스트레이션을 넣지 않는다
- **Domain/Infrastructure 계층**: 고객/어드민 공유. 분리하지 않는다
- **인증 필터**: URL prefix 기반으로 고객 인증 필터와 어드민 인증 필터를 분리 적용

### 모듈 분리가 의미 있어지는 시점

- 어드민 전용 도메인 로직이 복잡해지거나
- 고객/어드민 앱의 독립 배포가 필요해지거나
- 트래픽 패턴이 달라 별도 스케일링이 필요해질 때
