# DEC-003: 타인 주문 조회 시 응답 코드

| 항목 | 내용 |
|------|------|
| **상태** | OPEN |
| **관련 도메인** | Orders |
| **관련 요구사항** | UC-O03 주문 상세 조회 (Customer) |

## 기능적 요구사항

Customer가 다른 사용자의 주문을 조회하려 할 때의 HTTP 응답 코드를 결정해야 한다.

## 미결정 사항

| 선택지 | 설명 | 장점 | 단점 |
|--------|------|------|------|
| **403 FORBIDDEN** | 권한 없음을 명시적으로 반환 | 클라이언트에 명확한 피드백 | 주문 ID를 통해 타인의 주문 존재 여부를 추론 가능 (정보 노출) |
| **404 NOT_FOUND** | 주문이 없는 것처럼 반환 | 주문 존재 여부 비노출, 보안 강화 | 디버깅 시 실제 미존재와 권한 부족 구분 어려움 |

## 진짜 필요한 것

주문 ID를 통해 타인의 주문 존재 여부를 **추론할 수 없도록** 하는 보안 정책. 정보 노출 수준에 대한 보안 기준.

핵심 질문:
- 주문 ID가 순차적(auto-increment)인가? → 순차적이면 403 반환 시 전수 조사로 주문 건수 추론 가능
- 서비스의 보안 요구 수준은? (B2C 일반 커머스 vs 금융/의료)

## 변경 가능성

- B2B 확장 시 같은 조직의 주문을 조회할 수 있어야 할 수 있음
- 관리자(Admin)용 API는 이미 별도로 존재하므로 Customer API에서의 보안 우선이 가능
