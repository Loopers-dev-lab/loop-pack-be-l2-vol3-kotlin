# Round 3 재사용 Skills 추천

> 4개 도메인(Brand, Product, Like, Order)을 구현할 때 반복되는 작업 패턴을 식별하고,
> Claude Code Skill로 자동화할 수 있는 항목을 정리한다.

---

## 반복 패턴 분석

4개 도메인 구현 시 다음 작업이 **동일한 구조**로 반복된다:

| # | 반복 작업 | 반복 횟수 | 비고 |
|---|----------|-----------|------|
| 1 | JPA Entity 생성 (BaseEntity 상속, init 검증, protected set) | 5회 | Brand, Product, Like, Order, OrderItem |
| 2 | Repository 인터페이스 정의 (도메인 계층) | 4회 | Brand, Product, Like, Order |
| 3 | Repository 구현체 + JPA Repository 생성 (인프라 계층) | 4회 | 각 도메인별 |
| 4 | Domain Service 생성 (@Component, @Transactional) | 4회 | 각 도메인별 |
| 5 | Facade 생성 (Application Layer, 도메인 간 조합) | 4회 | Brand, Product, Like, Order |
| 6 | Info DTO 생성 (Facade ↔ Controller 간 데이터 전달) | 4회 | 각 도메인별 |
| 7 | Model 단위 테스트 (생성 검증, 비즈니스 메서드) | 5회 | 각 Entity별 |
| 8 | Service 단위 테스트 (MockK 기반) | 4회 | 각 도메인별 |
| 9 | Facade 단위 테스트 (MockK 기반) | 4회 | 각 도메인별 |

## 추천 Skills

### Skill 1: `/domain-entity` — 도메인 엔티티 스캐폴딩

**용도**: 새로운 도메인 엔티티를 프로젝트 패턴에 맞춰 자동 생성

**입력 예시**:
```
/domain-entity Brand name:String description:String? logoUrl:String? status:BrandStatus=ACTIVE
```

**생성 산출물**:
1. `domain/{domain}/{Domain}Model.kt` — JPA Entity (BaseEntity 상속, init 검증, protected set)
2. `domain/{domain}/{Domain}ModelTest.kt` — 생성 검증 단위 테스트 (유효 생성, 각 필드별 실패 케이스)

**반복 제거 효과**:
- 5개 Entity 생성 시 동일한 boilerplate 코드를 매번 작성하는 수고를 제거
- init 블록 검증 패턴, protected set 패턴, BaseEntity 상속 패턴을 일관되게 유지
- `@Column` 어노테이션, nullable 설정, length 제약을 필드 정의에서 자동 추론

**Skill 정의 (예시)**:
```yaml
name: domain-entity
description: 프로젝트 패턴에 맞는 JPA Entity와 단위 테스트를 자동 생성합니다
trigger: /domain-entity
args: "<도메인명> <필드:타입=기본값> ..."
steps:
  1. 기존 MemberModel.kt 패턴 분석
  2. 도메인명으로 패키지 경로 결정
  3. Entity 클래스 생성 (BaseEntity 상속, init 검증)
  4. 각 필드의 검증 테스트 생성 (유효 입력, 빈값, 범위 초과 등)
```

### Skill 2: `/domain-repository` — Repository 인터페이스 + 구현체 스캐폴딩

**용도**: 도메인 Repository 인터페이스와 Infrastructure 구현체를 한 번에 생성

**입력 예시**:
```
/domain-repository Brand
```

**생성 산출물**:
1. `domain/{domain}/{Domain}Repository.kt` — Repository 인터페이스 (기본 CRUD 메서드)
2. `infrastructure/{domain}/{Domain}JpaRepository.kt` — Spring Data JPA Repository
3. `infrastructure/{domain}/{Domain}RepositoryImpl.kt` — 구현체 (JPA Repository 위임)

**반복 제거 효과**:
- 4개 도메인에서 반복되는 3개 파일 세트(인터페이스 + JPA + 구현체)를 자동 생성
- `findByIdAndDeletedAtIsNull`, `save`, `findAll` 등 공통 메서드를 기본 포함
- DIP 패턴(도메인에 인터페이스, 인프라에 구현체)을 구조적으로 강제

**Skill 정의 (예시)**:
```yaml
name: domain-repository
description: Repository 인터페이스(Domain)와 구현체(Infrastructure)를 자동 생성합니다
trigger: /domain-repository
args: "<도메인명>"
steps:
  1. 기존 MemberRepository 패턴 분석
  2. Domain Layer에 인터페이스 생성 (기본 CRUD 메서드)
  3. Infrastructure Layer에 JPA Repository + 구현체 생성
  4. 소프트 삭제 패턴(deletedAtIsNull) 자동 적용
```

### Skill 3: `/domain-service` — 도메인 서비스 스캐폴딩

**용도**: 도메인 서비스와 단위 테스트를 프로젝트 패턴에 맞춰 생성

**입력 예시**:
```
/domain-service Brand --methods "create,findById,findAll,update,delete"
```

**생성 산출물**:
1. `domain/{domain}/{Domain}Service.kt` — @Component, 각 메서드에 @Transactional
2. `domain/{domain}/{Domain}ServiceTest.kt` — MockK 기반 단위 테스트 (정상/예외 시나리오)

**반복 제거 효과**:
- 4개 도메인의 서비스 + 테스트 파일을 일관된 패턴으로 생성
- `@Component`, `@Transactional`, `CoreException` 처리 등 공통 패턴 자동 적용
- MockK 기반 테스트 setup(every, verify, assertThat) 보일러플레이트 제거

---

### Skill 4: `/domain-facade` — Application Layer Facade 스캐폴딩

**용도**: Facade + Info DTO + 단위 테스트를 프로젝트 패턴에 맞춰 생성

**입력 예시**:
```
/domain-facade Product --depends "ProductService,BrandService"
```

**생성 산출물**:
1. `application/{domain}/{Domain}Facade.kt` — @Component, 의존 서비스 주입
2. `application/{domain}/{Domain}Info.kt` — 정보 전달 객체 (companion object의 from/of 메서드)
3. `application/{domain}/{Domain}FacadeTest.kt` — MockK 기반 단위 테스트

**반복 제거 효과**:
- 4개 도메인의 Facade + Info + Test 세트를 일관되게 생성
- `companion object { fun from(model): Info }` 패턴 자동 적용
- 의존 서비스 목록에 따라 Mock 설정 자동 생성

### Skill 5: `/domain-full-stack` — 도메인 전체 레이어 일괄 스캐폴딩

**용도**: 하나의 도메인에 대해 4개 레이어(Domain → Infrastructure → Application)를 한 번에 생성

**입력 예시**:
```
/domain-full-stack Like userId:Long productId:Long --methods "like,unlike,findByUserId"
```

**생성 산출물** (총 8개 파일):
1. `domain/like/LikeModel.kt`
2. `domain/like/LikeRepository.kt`
3. `domain/like/LikeService.kt`
4. `infrastructure/like/LikeJpaRepository.kt`
5. `infrastructure/like/LikeRepositoryImpl.kt`
6. `application/like/LikeFacade.kt`
7. `application/like/LikeInfo.kt`
8. 테스트 파일 3개 (Model, Service, Facade)

**반복 제거 효과**:
- 도메인 하나를 추가할 때 필요한 모든 파일을 일관된 구조로 한 번에 생성
- 수동 생성 시 발생하는 파일 누락, 패턴 불일치, import 오류를 방지
- 생성 후 비즈니스 로직만 채우면 되므로 구조 작업에 소비되는 시간 절감

---

### Skill 6: `/domain-model-test` — 도메인 모델 단위 테스트 자동 생성

**용도**: Entity의 필드 정의를 분석하여 검증 테스트를 자동 생성

**입력 예시**:
```
/domain-model-test ProductModel
```

**생성 산출물**:
1. `domain/product/ProductModelTest.kt` — 모든 검증 규칙에 대한 단위 테스트

**자동 생성되는 테스트 케이스**:
- `String` 필드 → 빈값, 길이 초과 테스트
- `Long/Int` 필드 (>= 0 제약) → 음수 테스트
- `Enum` 필드 → 기본값 확인 테스트
- 비즈니스 메서드 → 정상/예외 시나리오 테스트

**반복 제거 효과**:
- 5개 Entity에 대해 검증 테스트를 자동 생성 (평균 5~10개 테스트 케이스/Entity)
- Arrange-Act-Assert 패턴, @DisplayName, @Nested 구조를 일관되게 적용
- 테스트 누락(경계값, 예외 케이스)을 구조적으로 방지

## 우선순위 추천

| 순위 | Skill | 효과 | 구현 난이도 |
|------|-------|------|-----------|
| 1 | `/domain-full-stack` | 도메인 하나당 11개 파일 자동 생성, 가장 큰 시간 절감 | 높음 |
| 2 | `/domain-entity` | Entity + 테스트 생성, 가장 빈번한 반복 작업 | 중간 |
| 3 | `/domain-repository` | DIP 패턴 강제, 3개 파일 세트 자동 생성 | 낮음 |
| 4 | `/domain-model-test` | 검증 테스트 자동화, TDD RED 단계 가속 | 중간 |
| 5 | `/domain-service` | 서비스 + 테스트 생성 | 중간 |
| 6 | `/domain-facade` | Facade + Info + 테스트 생성 | 중간 |

### 실용적 추천

**최소 구현**: `/domain-entity` + `/domain-repository`를 먼저 만들면,
가장 반복적인 boilerplate 코드 생성(Entity 5회 + Repository 세트 4회)을 자동화할 수 있다.

**최대 효과**: `/domain-full-stack`을 구현하면 도메인 하나를 추가할 때
구조 작업 없이 바로 비즈니스 로직 구현에 집중할 수 있다.
