```markdown
## 도메인 & 객체 설계 전략

- 도메인 객체는 비즈니스 규칙을 캡슐화한다. 데이터를 들고만 있는 빈껍데기 모델은 지양한다.
- 규칙이 여러 서비스에 반복적으로 나타나면, 그 규칙은 도메인 객체에 속할 가능성이 높다.
- Entity는 고유 ID로 식별되며, 상태 변화가 중요한 객체이다 (예: Product, Order).
- Value Object는 값 자체가 의미이며, 불변(immutable)이다 (예: Money, Quantity).
- Domain Service는 특정 Entity에 속하기 어려운 도메인 로직을 처리하며, 상태를 갖지 않는다.
- 비즈니스 의미가 커질 수 있는 개념은 독립된 도메인 단위로 격리한다
  (예: 좋아요는 Product.likedUserIds가 아닌 독립된 Like 도메인).
- 각 기능에 대한 책임과 결합도에 대해 개발자의 의도를 확인하고 개발을 진행한다.
```

```markdown
- 레이어드 아키텍처를 따르며, DIP(의존성 역전 원칙)을 준수한다.
```

### BrandModel 설계

```kotlin
@Entity
@Table(name = "brand")
class BrandModel(
    name: String,
    description: String? = null,
    logoUrl: String? = null,
    status: BrandStatus = BrandStatus.ACTIVE,
) : BaseEntity() {

    @Column(nullable = false, unique = true, length = 100)
    var name: String = name
        protected set

    @Column(length = 500)
    var description: String? = description
        protected set

    @Column(name = "logo_url", length = 500)
    var logoUrl: String? = logoUrl
        protected set

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 20)
    var status: BrandStatus = status
        protected set

    init {
        validateName(name)
    }

    // 비즈니스 메서드
    fun update(name: String, description: String?, logoUrl: String?, status: BrandStatus) { ... }
    fun isActive(): Boolean = status == BrandStatus.ACTIVE

    // 검증 로직
    private fun validateName(name: String) {
        if (name.isBlank()) throw CoreException(ErrorType.BAD_REQUEST, "브랜드명은 필수입니다.")
        if (name.length > 100) throw CoreException(ErrorType.BAD_REQUEST, "브랜드명은 100자 이하여야 합니다.")
    }
}
```

## Phase 2: Product 도메인 모델링

### 목표

상품을 독립된 도메인으로 모델링한다.
상품은 재고를 자체적으로 관리하며, 재고 차감/증가의 도메인 규칙을 엔티티 내부에서 처리한다.
좋아요 수(likesCount)는 상품이 보유하지만, 좋아요 등록/취소는 Like 도메인이 관리한다.

### 도메인 개념 정리

| 개념 | 분류 | 근거 |
|------|------|------|
| `ProductModel` | Entity | 고유 ID로 식별, 재고/가격/상태 변화가 중요 |
| `SaleStatus` | Value Object (Enum) | 판매 상태를 표현하는 불변 값 |
| `DisplayStatus` | Value Object (Enum) | 노출 상태를 표현하는 불변 값 |

### 파일 구조

```
domain/product/
  ├── ProductModel.kt       — 상품 JPA 엔티티 (BaseEntity 상속)
  ├── SaleStatus.kt         — 판매 상태 enum (SELLING, STOP_SELLING)
  ├── DisplayStatus.kt      — 노출 상태 enum (VISIBLE, HIDDEN)
  ├── ProductRepository.kt  — Repository 인터페이스
  └── ProductService.kt     — 도메인 서비스
```

### ProductModel 설계

```kotlin
@Entity
@Table(name = "product")
class ProductModel(
    name: String,
    price: Long,
    brandId: Long,
    description: String? = null,
    thumbnailImageUrl: String? = null,
    stockQuantity: Int = 0,
    likesCount: Long = 0,
    saleStatus: SaleStatus = SaleStatus.SELLING,
    displayStatus: DisplayStatus = DisplayStatus.VISIBLE,
) : BaseEntity() {

    // ... 필드 정의 (protected set)

    init {
        validateName(name)
        validatePrice(price)
        validateStockQuantity(stockQuantity)
        validateLikesCount(likesCount)
    }

    // === 재고 관리 (도메인 규칙) ===

    /** 재고를 차감한다. 재고가 부족하면 도메인 예외를 발생시킨다. */
    fun decreaseStock(quantity: Int) {
        require(quantity > 0) { "차감 수량은 1 이상이어야 합니다." }
        if (this.stockQuantity < quantity) {
            throw CoreException(
                ErrorType.BAD_REQUEST,
                "상품의 재고가 부족합니다. (상품명: $name, 요청 수량: ${quantity}개, 현재 재고: ${stockQuantity}개)"
            )
        }
        this.stockQuantity -= quantity
    }

    /** 재고를 증가시킨다. */
    fun increaseStock(quantity: Int) {
        require(quantity > 0) { "증가 수량은 1 이상이어야 합니다." }
        this.stockQuantity += quantity
    }

    // === 좋아요 수 관리 ===

    fun incrementLikesCount() { this.likesCount++ }
    fun decrementLikesCount() {
        if (this.likesCount > 0) this.likesCount--
    }

    // === 검증 로직 ===

    private fun validateName(name: String) { ... }
    private fun validatePrice(price: Long) {
        if (price < 0) throw CoreException(ErrorType.BAD_REQUEST, "가격은 0 이상이어야 합니다.")
    }
    private fun validateStockQuantity(stockQuantity: Int) {
        if (stockQuantity < 0) throw CoreException(ErrorType.BAD_REQUEST, "재고 수량은 0 이상이어야 합니다.")
    }
    private fun validateLikesCount(likesCount: Long) {
        if (likesCount < 0) throw CoreException(ErrorType.BAD_REQUEST, "좋아요 수는 0 이상이어야 합니다.")
    }
}
```

### 설계 의사결정: 재고 차감을 Entity에 둔 이유

재고 음수 방지는 **상품 자체의 비즈니스 규칙**이다.
외부 서비스(OrderService 등)가 `product.stockQuantity -= quantity`처럼 직접 조작하면
검증 로직이 흩어지고, 음수 재고가 발생할 수 있다.
`decreaseStock(quantity)` 메서드를 통해 **항상 검증을 거치도록 캡슐화**한다.

## Phase 3: Like 도메인 모델링

### 목표

좋아요를 유저-상품 간의 관계로서 **독립된 도메인**으로 모델링한다.
좋아요 등록/취소 시 멱등성을 보장하고, 상품의 likesCount를 연동한다.

### 도메인 개념 정리

| 개념 | 분류 | 근거 |
|------|------|------|
| `LikeModel` | Entity | 고유 ID로 식별, 누가 언제 어떤 상품을 좋아했는지 추적 |

### 설계 의사결정: 좋아요를 독립 도메인으로 분리한 이유

좋아요를 `Product.likedUserIds`에 내장하면:
- 상품 조회 시 불필요한 유저 ID 목록이 로드된다
- 좋아요 이력 추적(언제 눌렀는지, 취소 이력)이 불가능하다
- 유저별 좋아요 목록 조회가 비효율적이다

독립 도메인으로 분리하면:
- 좋아요 자체의 생명주기(등록/취소/복원)를 관리할 수 있다
- 유저별, 상품별 조회가 유연하다
- 향후 좋아요 기반 추천, 인기 상품 정렬 등 확장이 용이하다

### 파일 구조

```
domain/like/
  ├── LikeModel.kt       — 좋아요 JPA 엔티티 (BaseEntity 상속)
  ├── LikeRepository.kt  — Repository 인터페이스
  └── LikeService.kt     — 도메인 서비스 (등록/취소/조회)
```

### LikeModel 설계

```kotlin
@Entity
@Table(
    name = "likes",
    uniqueConstraints = [UniqueConstraint(columnNames = ["user_id", "product_id"])]
)
class LikeModel(
    userId: Long,
    productId: Long,
) : BaseEntity() {

    @Column(name = "user_id", nullable = false)
    var userId: Long = userId
        protected set

    @Column(name = "product_id", nullable = false)
    var productId: Long = productId
        protected set
}
```

### LikeService 설계

```kotlin
@Component
class LikeService(
    private val likeRepository: LikeRepository,
) {
    /**
     * 좋아요를 등록한다. 멱등성을 보장한다.
     * - 기록 없음 → 새로 생성
     * - 삭제된 기록 있음 → restore()로 복원
     * - 활성 기록 있음 → 아무 작업 없이 반환
     */
    @Transactional
    fun like(userId: Long, productId: Long): LikeModel { ... }

    /**
     * 좋아요를 취소한다. 멱등성을 보장한다.
     * - 활성 기록 있음 → delete()로 소프트 삭제
     * - 기록 없음 또는 이미 삭제됨 → 아무 작업 없이 반환
     */
    @Transactional
    fun unlike(userId: Long, productId: Long) { ... }

    /** 유저의 활성 좋아요 목록을 조회한다. */
    @Transactional(readOnly = true)
    fun findByUserId(userId: Long, pageable: Pageable): Page<LikeModel> { ... }
}
```

## Phase 4: Order 도메인 모델링

### 목표

주문을 Entity + OrderItem으로 모델링한다.
주문 생성 시 상품 재고 차감, 스냅샷 저장, 원자성 보장을 도메인 레벨에서 처리한다.

### 도메인 개념 정리

| 개념 | 분류 | 근거 |
|------|------|------|
| `OrderModel` | Entity | 고유 ID로 식별, 주문 상태 변화가 중요 (ORDERED 등) |
| `OrderItemModel` | Entity | 주문 항목별 스냅샷(상품명, 가격, 브랜드명) 보존, 개별 식별 필요 |
| `OrderStatus` | Value Object (Enum) | 주문 상태를 표현하는 불변 값 |

### 설계 의사결정: 스냅샷을 OrderItem에 저장하는 이유

주문 후 상품 정보(이름, 가격)가 변경되거나 상품이 삭제되어도,
주문 시점의 정보를 보존해야 정확한 주문 이력을 제공할 수 있다.
`OrderItemModel`에 상품명, 가격, 브랜드명을 **비정규화(denormalize)** 하여 스냅샷을 저장한다.

### 파일 구조

```
domain/order/
  ├── OrderModel.kt       — 주문 JPA 엔티티 (BaseEntity 상속)
  ├── OrderItemModel.kt   — 주문 항목 JPA 엔티티 (스냅샷 포함)
  ├── OrderStatus.kt      — 주문 상태 enum (ORDERED)
  ├── OrderRepository.kt  — Repository 인터페이스
  └── OrderService.kt     — 주문 도메인 서비스
```

### OrderModel 설계

```kotlin
@Entity
@Table(name = "orders")
class OrderModel(
    userId: Long,
    orderStatus: OrderStatus = OrderStatus.ORDERED,
) : BaseEntity() {

    @Column(name = "user_id", nullable = false)
    var userId: Long = userId
        protected set

    @Enumerated(EnumType.STRING)
    @Column(name = "order_status", nullable = false, length = 20)
    var orderStatus: OrderStatus = orderStatus
        protected set

    @Column(name = "total_amount", nullable = false)
    var totalAmount: Long = 0
        protected set

    @OneToMany(mappedBy = "order", cascade = [CascadeType.ALL], orphanRemoval = true)
    val orderItems: MutableList<OrderItemModel> = mutableListOf()

    /** 주문 항목을 추가하고 총 금액을 재계산한다. */
    fun addItem(item: OrderItemModel) {
        orderItems.add(item)
        recalculateTotalAmount()
    }

    private fun recalculateTotalAmount() {
        this.totalAmount = orderItems.sumOf { it.subTotal }
    }
}
```

### OrderItemModel 설계

```kotlin
@Entity
@Table(name = "order_item")
class OrderItemModel(
    order: OrderModel,
    productId: Long,
    productName: String,    // 스냅샷
    brandName: String,      // 스냅샷
    price: Long,            // 스냅샷
    quantity: Int,
) : BaseEntity() {

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id", nullable = false)
    var order: OrderModel = order
        protected set

    // ... 스냅샷 필드들 (protected set)

    @Column(name = "sub_total", nullable = false)
    var subTotal: Long = price * quantity
        protected set

    init {
        validateQuantity(quantity)
    }

    private fun validateQuantity(quantity: Int) {
        if (quantity < 1 || quantity > 99) {
            throw CoreException(ErrorType.BAD_REQUEST, "주문 수량은 1개 이상 99개 이하여야 합니다.")
        }
    }
}
```

### OrderService 설계

```kotlin
@Component
class OrderService(
    private val orderRepository: OrderRepository,
) {
    /**
     * 주문을 생성한다.
     * - 상품 중복 검증
     * - 재고 확인 및 차감 (Product.decreaseStock)
     * - 스냅샷 저장
     * - 원자성 보장 (일부 재고 부족 시 전체 실패)
     *
     * 주의: 재고 차감은 Product Entity의 decreaseStock()에 위임한다.
     * OrderService는 흐름을 조율할 뿐, 재고 검증 로직 자체를 갖지 않는다.
     */
    @Transactional
    fun createOrder(
        userId: Long,
        orderItems: List<Pair<ProductModel, Int>>,  // (상품, 수량)
        brandNameResolver: (Long) -> String,         // brandId → brandName 해석
    ): OrderModel {
        // 1. 주문 생성
        val order = OrderModel(userId = userId)

        // 2. 각 상품의 재고 차감 + 스냅샷 생성
        orderItems.forEach { (product, quantity) ->
            product.decreaseStock(quantity)  // 재고 부족 시 여기서 예외 발생 → 트랜잭션 롤백
            val item = OrderItemModel(
                order = order,
                productId = product.id,
                productName = product.name,
                brandName = brandNameResolver(product.brandId),
                price = product.price,
                quantity = quantity,
            )
            order.addItem(item)
        }

        // 3. 저장
        return orderRepository.save(order)
    }
}
```

### 설계 의사결정: 재고 차감 책임 분배

| 관심사 | 책임 주체 | 이유 |
|--------|----------|------|
| 재고가 충분한지 검증, 재고 차감 | `ProductModel.decreaseStock()` | 재고는 상품의 비즈니스 규칙이다 |
| 주문 항목 조합, 스냅샷 생성 | `OrderService.createOrder()` | 주문 생성 흐름 조율은 Order 도메인의 책임이다 |
| 상품 조회, 브랜드 조회, 인증 검증 | `OrderFacade` (Phase 5) | 도메인 간 조합은 Application Layer의 책임이다 |

## Phase 5: Application Layer 통합

### 목표

Application Layer(Facade)에서 여러 도메인을 경량으로 조합하여 유스케이스를 완성한다.
Facade는 비즈니스 로직을 직접 구현하지 않고, 도메인 서비스에 위임한다.

### 핵심 유스케이스

#### 5.1 상품 상세 조회 (Product + Brand + Like 조합)

```
ProductFacade.getProductDetail(productId)
  → ProductService.findById(productId)          // 상품 조회
  → BrandService.findById(product.brandId)       // 브랜드 조회
  → 상품 정보 + 브랜드 이름 + likesCount 조합
  → ProductInfo 반환
```

```kotlin
@Component
class ProductFacade(
    private val productService: ProductService,
    private val brandService: BrandService,
) {
    fun getProductDetail(productId: Long): ProductInfo {
        val product = productService.findById(productId)
        val brand = brandService.findById(product.brandId)
        return ProductInfo.of(product, brand)
    }

    fun getProductList(brandId: Long?, sort: String, pageable: Pageable): Page<ProductInfo> {
        val products = productService.findAll(brandId, sort, pageable)
        // 브랜드 정보를 일괄 조회하여 N+1 문제 방지
        val brandIds = products.content.map { it.brandId }.distinct()
        val brandMap = brandService.findAllByIds(brandIds).associateBy { it.id }
        return products.map { product ->
            ProductInfo.of(product, brandMap[product.brandId]!!)
        }
    }
}
```

#### 5.2 좋아요 등록/취소 (Like + Product 조합)

```
LikeFacade.likeProduct(userId, productId)
  → ProductService.findById(productId)    // 상품 존재 검증
  → LikeService.like(userId, productId)   // 좋아요 등록 (멱등)
  → ProductService.incrementLikesCount(productId)  // likesCount 증가 (신규 등록 시에만)
```

```kotlin
@Component
class LikeFacade(
    private val likeService: LikeService,
    private val productService: ProductService,
) {
    @Transactional
    fun likeProduct(userId: Long, productId: Long) {
        productService.findById(productId)  // 상품 존재 검증 (삭제된 상품이면 NOT_FOUND)
        val isNewLike = likeService.like(userId, productId)
        if (isNewLike) {
            productService.incrementLikesCount(productId)
        }
    }

    @Transactional
    fun unlikeProduct(userId: Long, productId: Long) {
        val wasActive = likeService.unlike(userId, productId)
        if (wasActive) {
            productService.decrementLikesCount(productId)
        }
    }
}
```

#### 5.3 주문 생성 (Order + Product + Brand 조합)

```
OrderFacade.createOrder(userId, items)
  → 입력 검증 (중복 상품, 수량 범위)
  → ProductService.findAllByIds(productIds)     // 상품 일괄 조회
  → BrandService.findById(brandId)              // 브랜드 조회 (스냅샷용)
  → OrderService.createOrder(userId, orderItems, brandNameResolver)
```

```kotlin
@Component
class OrderFacade(
    private val orderService: OrderService,
    private val productService: ProductService,
    private val brandService: BrandService,
) {
    @Transactional
    fun createOrder(userId: Long, items: List<OrderItemRequest>): OrderInfo {
        // 1. 입력 검증
        validateNoDuplicateProducts(items)

        // 2. 상품 일괄 조회
        val productIds = items.map { it.productId }
        val productMap = productService.findAllByIdsForUpdate(productIds)
            .associateBy { it.id }

        // 3. 존재 확인
        items.forEach { item ->
            productMap[item.productId]
                ?: throw CoreException(ErrorType.NOT_FOUND, "존재하지 않는 상품입니다.")
        }

        // 4. 브랜드 일괄 조회 (스냅샷용)
        val brandIds = productMap.values.map { it.brandId }.distinct()
        val brandMap = brandService.findAllByIds(brandIds).associateBy { it.id }

        // 5. 주문 생성 (재고 차감은 OrderService → Product.decreaseStock에서 처리)
        val orderItems = items.map { item ->
            val product = productMap[item.productId]!!
            product to item.quantity
        }
        val order = orderService.createOrder(
            userId = userId,
            orderItems = orderItems,
            brandNameResolver = { brandId -> brandMap[brandId]?.name ?: "" }
        )

        return OrderInfo.from(order)
    }
}
```

### 도메인 간 직접 의존 관계

| From | To | 이유 |
|------|----|------|

### 통합 테스트 (Testcontainers 기반)

- **Repository 테스트**: 실제 DB에 저장/조회 동작 검증
- **E2E 테스트**: 전체 API 흐름 검증 (Controller → Facade → Service → Repository)

## Round 3 체크리스트 매핑

### Product / Brand 도메인

| 체크리스트 항목 | 구현 위치 |
|---------------|---------|
| 상품 정보 객체는 브랜드 정보, 좋아요 수를 포함한다 | `ProductFacade.getProductDetail()` → `ProductInfo` |
| 정렬 조건(latest, price_asc, likes_desc) 조회 설계 | `ProductRepository` + `ProductService.findAll()` |
| 상품은 재고를 가지고 있고, 주문 시 차감 가능 | `ProductModel.stockQuantity` + `decreaseStock()` |
| 재고 음수 방지는 도메인 레벨에서 처리 | `ProductModel.decreaseStock()` 내부 검증 |

### Like 도메인

| 체크리스트 항목 | 구현 위치 |
|---------------|---------|
| 좋아요는 유저-상품 간 관계로 별도 도메인 분리 | `domain/like/LikeModel.kt` |
| 좋아요 수는 상품 상세/목록에서 제공 | `ProductModel.likesCount` + `ProductFacade` |
| 좋아요 등록/취소 흐름 단위 테스트 | `LikeServiceTest`, `LikeFacadeTest` |

### Order 도메인

| 체크리스트 항목 | 구현 위치 |
|---------------|---------|
| 여러 상품 포함, 각 상품 수량 명시 | `OrderModel` + `OrderItemModel` |
| 주문 시 재고 차감 | `OrderFacade` → `ProductModel.decreaseStock()` |
| 재고 부족 예외 흐름 설계 | `ProductModel.decreaseStock()` 예외 → 트랜잭션 롤백 |
| 정상/예외 주문 단위 테스트 | `OrderServiceTest`, `OrderFacadeTest` |

### 도메인 서비스

| 체크리스트 항목 | 구현 위치 |
|---------------|---------|
| 도메인 내부 규칙은 Domain Service에 위치 | `BrandService`, `ProductService`, `LikeService`, `OrderService` |
| Product + Brand 조합은 Application Layer | `ProductFacade.getProductDetail()` |
| 복합 유스케이스는 Application Layer | `OrderFacade.createOrder()` |
| Domain Service는 상태 없이 협력 중심 설계 | 모든 Service는 `@Component`, 상태 없음 |

### 소프트웨어 아키텍처 & 설계

| 체크리스트 항목 | 구현 위치 |
|---------------|---------|
| Application → Domain ← Infrastructure | 패키지 구조 + 의존성 방향 |
| Application Layer의 orchestration | `ProductFacade`, `LikeFacade`, `OrderFacade` |
| Entity, VO, Domain Service에 비즈니스 로직 | 각 도메인 Model + Service |
| Repository Interface → Domain, 구현체 → Infra | `domain/*/Repository.kt` → `infrastructure/*/RepositoryImpl.kt` |
| 계층 + 도메인 기준 패키지 | `/domain/order`, `/application/like` 등 |
| Fake/Stub 기반 단위 테스트 | MockK 기반 Service/Facade 테스트 |
